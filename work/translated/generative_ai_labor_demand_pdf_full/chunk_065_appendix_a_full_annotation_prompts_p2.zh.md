【中文译文】
# 附录A 完整注释提示

任务提取规则：
- 任务必须基于正文文本，并反映所提及的实际动作、工具、系统、材料、交互或工作环境。
- 尽可能保留具体信息（软件名称、设备、文档、样品、客户、患者、系统、环境）。
- 关注可观察的动作（明确动词+宾语）。
- 仅包含正文文本中明确陈述或清晰暗示的任务。
严格避免：
- 公司信息、薪酬、福利、资格要求或招聘说明。
-------------------------------------------------第3步：将任务匹配到技能组
-------------------------------------------------
为每个任务分配唯一的 skill_group_id。
- 将任务与所有技能组（专业组+通用组）进行比对。
- 选择语义最接近的匹配。
- 若出现平局，优先选择专业组（S*）而非通用组（C*）。
- 若不存在任何技能，则将所有任务分配给 NS0。
-------------------------------------------------输出格式（严格——仅 JSON）
-------------------------------------------------
返回严格符合以下结构的一个 JSON 对象：
{
  "posting_id": "<ID>",
  "posting_title": "<标题名称>",
  "skills_groups": [
    {"group_id": "S1", "group_skills": ["..."]},
    {"group_id": "C1", "group_skills": ["..."]},
    {"group_id": "NS0", "group_skills": []}
  ],
  "tasks": [
    {
      "task_id": "t1",
      "task": "<8–50词的职位特定任务描述>",
      "skill_group_id": "S1 | C1 | NS0"
    }
  ]
}
最终检查：
- 仅有效的 JSON。
- 3–10个任务。
- 任务反映的是职位描述中具体的工作内容，而非广义的角色描述。
- 每个任务有且仅有一个 skill_group_id。

A.2 第二阶段提示：暴露分类
你是生成式AI领域的专家。你的任务是根据工作任务受生成式AI/大语言模型工具影响的程度对其进行分类。
假设：
- 该角色中具有平均专业水平的工人。
- 可以使用强大LLM和现成的生成式AI工具。
- 可以使用标准笔记本电脑工具（例如麦克风、扬声器）。
- 除笔记本电脑外，无物理工具或物理在场要求。

- skill group: 技能组
- skill_group_id: skill_group_id（保留英文）
- posting_id: posting_id（保留英文）
- posting_title: posting_title（保留英文）
- group_id: group_id（保留英文）

- S1, C1: S1, C1（保留英文）

【本段要点】
- 给出了任务提取、技能组匹配和JSON输出格式的严格规则。
- 强调任务必须基于正文，避免无关信息；匹配时优先选择专业组。
- 第二阶段提示要求基于生成式AI/LLM能力对工作任务进行暴露分类，并设定了工人能力和工具的假设条件。

【可能需人工复核】
- “posting”一词在中文语境中可能有“招聘信息”“职位描述”“岗位公告”等译法，此处统一采用“职位描述”或“招聘信息”均合理，但需确保全篇一致。
- “exposure”在AI暴露语境下常译为“暴露度”或“受影响程度”，此处译为“暴露”可能需根据后续上下文确认是否调整。
- “skill_group_id”“posting_id”等JSON键名保留英文，但读者是否理解需评估。
- 原文中“A.2 Stage 2 Prompt”在附录中，标题格式需与整体附录结构一致。
