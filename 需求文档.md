## 一、项目概述

### 1.1 项目背景

本项目结合人工智能导论课程的知识体系，设计并实现一个**学习助手 Agent + RAG（检索增强生成）系统**。系统面向课程学习场景，支持用户上传多种格式的课程资料作为知识库，并基于知识库自动构建知识图谱。用户通过与 Agent 对话的方式提出学习问题，Agent 结合知识库检索结果和知识图谱中的实体关系进行回答，并展示回答的依据来源。

### 1.2 项目目标

- 构建一个本地化的文档知识库管理系统，支持多种文档格式上传与解析
- 基于知识库自动构建知识图谱，建立文档片段与图谱实体之间的双向索引映射
- 实现一个多轮对话 Agent，能够结合知识库和知识图谱进行 RAG 问答
- 提供简单可用的管理平台，第一期以单用户本地部署为主

## 二、文档管理模块

### 2.1 支持的文件格式

| 格式 | 说明 | 备注 |
|------|------|------|
| PDF | Portable Document Format | 最主要的课程资料格式 |
| DOC / DOCX | Microsoft Word 文档 | 旧版和新版 Word 格式 |
| PPT / PPTX | Microsoft PowerPoint 演示文稿 | 幻灯片内容提取 |
| XLSX | Microsoft Excel 电子表格 | 表格数据提取 |
| MD | Markdown 格式文档 | 纯文本，解析简单 |
| TXT | 纯文本文件 | 最基础的文本格式 |
| HTML | 网页格式文档 | 支持 HTML 文件解析 |

### 2.2 文件上传规格

- **单文件大小限制**：20MB 以内
- **上传方式**：
  - 单文件上传：一次上传一个文件
  - 批量上传：一次选择并上传多个文件
- **上传后处理**：文件存储至本地文件系统，建立基础索引元数据

### 2.3 文档操作

| 操作 | 说明 |
|------|------|
| 查看列表 | 展示已上传文档的名称、格式、上传时间等信息 |
| 新增上传 | 上传新的文档至知识库 |
| 删除 | 删除指定文档，同步清理关联的索引和图谱数据 |
| 修改 | 简单版本暂不做复杂的文档修改功能 |

### 2.4 版本管理

**第一期不做版本管理**，后续版本可考虑实现文档更新历史版本保留功能。

### 2.5 文档解析与分块

**解析策略（简单版本）**：

- 对不同格式的文档进行文本内容提取
- 统一转换为纯文本格式存储
- 按固定长度进行分块（Chunking）
  - 每块字数：500 字
  - 相邻块重叠字数：50 字（保证上下文连贯性）
- 记录每个 Chunk 的来源文档和位置信息

**暂不实现的功能（后期版本）**：

- OCR 识别（扫描版 PDF 的文字提取）
- 复杂语义分块策略
- 表格、多媒体内容的特殊处理

## 三、知识库模块

### 3.1 存储架构

```
knowledge_base/
├── documents/          # 原始文档存储
│   ├── doc_001.pdf
│   ├── doc_002.docx
│   └── ...
├── chunks/             # 分块文本存储
│   ├── doc_001/
│   │   ├── chunk_000.txt
│   │   ├── chunk_001.txt
│   │   └── ...
│   └── ...
├── embeddings/         # 向量索引存储
│   ├── index.faiss
│   └── chunk_metadata.json
├── dictionary/         # 领域词典存储（实体抽取用）
│   └── domain_terms.json
└── config.yaml        # 知识库配置文件
```

### 3.2 向量索引

- **嵌入模型**：bge-m3（BAAI/智源开源模型）
  - 开源免费，可本地部署
  - 中英文语义支持优秀
  - 适合教育类文档的知识检索场景
- **索引工具**：FAISS（Facebook AI Similarity Search）
  - 高效的向量相似度检索
  - 支持大规模向量索引
  - 轻量易用，本地运行

### 3.3 索引管理

| 操作 | 说明 |
|------|------|
| 索引重建 | 用户手动点击按钮或调用接口，触发全量索引重建 |
| 增量更新 | **第一期暂不做**，新增文档后统一通过全量重建处理 |
| 删除后同步 | 删除文档后，用户手动触发全量重建 |

**Top-K 检索配置**：
- 默认 Top-K = 5，即每次检索返回与查询最相似的 5 个 Chunk
- 该参数在配置文件中可调整（`retrieval.top_k`）
- 当文档规模较大时，可适当增大 K 值以提高召回率

### 3.4 检索流程

1. 用户提问或检索请求
2. 将查询文本通过 bge-m3 转换为向量
3. 在 FAISS 索引中进行相似度检索，返回 Top-K 个最相关的 Chunk
4. 合并检索结果，返回 Chunk 文本及其来源文档信息

### 3.5 扩展预留

以下功能在第一期中预留接口和数据结构设计空间，后期版本实现：

| 功能 | 说明 |
|------|------|
| 数据库持久化 | 引入 PostgreSQL 等关系数据库存储元数据 |
| 多用户隔离 | 实现用户级别的知识库隔离 |
| 全文索引 | 引入 Elasticsearch 等支持混合检索（向量+关键词） |
| 增量索引更新 | 新增文档时仅更新增量部分，不做全量重建 |

## 四、知识图谱模块

### 4.1 构建触发机制

- 用户手动点击管理平台中的「重建图谱」按钮
- 提供接口供外部调用：`POST /api/knowledgegraph/rebuild`
- 图谱构建独立于知识库索引，两者的重建操作互不影响

### 4.2 存储方式

| 存储类型 | 说明 | 后期升级路径 |
|----------|------|-------------|
| 内存存储 | 构建时缓存在内存中，供问答模块实时查询 | 迁移至图数据库 |
| 文件存储 | 将图谱数据持久化保存为 JSON / GraphML 文件 | 引入 Neo4j 等图数据库 |

**图谱文件存储结构**：

```
knowledge_graph/
├── entities.json      # 实体列表
├── relations.json     # 关系列表
├── mapping.json       # 双向索引映射（Chunk ID <-> 实体 ID）
└── visualization.png  # 图谱可视化图片
```

### 4.3 图谱粒度设计

**方案：命名实体节点 + 概念节点**

#### 4.3.1 节点类型

| 节点类型 | 说明 | 示例 |
|----------|------|------|
| 实体节点（Entity） | 文档中提取的具体命名实体 | 深度学习、CNN、孙圣力 |
| 概念节点（Concept） | 抽象的知识点和概念 | 机器学习、计算机视觉 |

#### 4.3.2 边（关系）类型

| 关系类型 | 标识 | 说明 | 示例 |
|----------|------|------|------|
| 上下位关系 | is-a | 概念之间的包含与归属 | 机器学习 is-a 人工智能 |
| 包含关系 | part-of | 部分与整体的关系 | 卷积层 part-of CNN |
| 因果关系 | cause-effect | 事件间的因果链条 | 梯度消失 cause-effect 长序列训练困难 |
| 相关关系 | related-to | 一般性语义关联 | 注意力机制 related-to Transformer |
| 定义关系 | definition-of | 概念与其定义的描述 | 深度学习 definition-of 多层神经网络 |

### 4.4 实体抽取策略

第一期采用**规则 + 领域词典**的实体抽取方法：

- 使用正则表达式提取特定模式的实体（如术语、专有名词、书名号内的概念等）
- 利用课程领域的关键词词典进行术语识别（词典由系统管理员维护，存储为 JSON/TXT 文件，支持用户导入扩充）
- 结合预训练语言模型的词汇分析能力辅助识别（中文字符级切分 + 停用词过滤）
- 实体去重与归一化：相同实体在不同文档中出现时合并为同一节点

### 4.5 图谱与知识库的双向索引映射

**目的**：建立文本 Chunk 与图谱实体之间的关联关系，实现问答时能够同时返回文档片段和图谱子图。

**映射结构（Chunk -> 实体）**：

```json
{
  "chunk_id": "doc_001_chunk_003",
  "entities": ["实体A", "实体B", "概念C"],
  "entity_ids": ["E001", "E002", "C001"]
}
```

**逆向索引（实体 -> Chunk）**：

```json
{
  "entity_id": "E001",
  "entity_name": "卷积神经网络",
  "chunks": ["doc_001_chunk_003", "doc_002_chunk_015"],
  "relations": [
    {"target": "E002", "type": "part-of"},
    {"target": "C001", "type": "is-a"}
  ]
}
```

### 4.6 图谱规模

**不设硬性上限**，由用户自行控制文档规模和图谱复杂度。系统应在图谱构建过程中记录节点和边的数量，当规模过大时给出性能提示。

### 4.7 图谱可视化

**方案：生成静态图片**

- 使用 Python 图形库（如 Graphviz、pyecharts）将图谱渲染为 PNG 图片
- 用户在管理平台点击「下载图谱」按钮获取图片
- 图片分辨率适中，支持基本的关系布局展示

**扩展方案（后期版本）**：

- 提供交互式 Web 可视化页面（使用 D3.js 或 ECharts）
- 支持图谱的缩放、拖拽、节点筛选等交互功能
- 支持导出为 GraphML、DOT 等标准图数据格式

### 4.8 扩展预留

| 功能 | 说明 |
|------|------|
| 图数据库集成 | 后期引入 Neo4j 等图数据库替代文件存储 |
| 增量图谱更新 | 新增文档时仅扩展相关子图，不做全量重建 |
| 动态图谱交互 | Web 端交互式图谱可视化 |

## 五、Agent 问答模块

### 5.1 对话模式

- **多轮对话**：Agent 能够记忆对话上下文，支持多轮连续问答
- **上下文管理**：保留最近 N 轮对话历史（如最近 10 轮），超出后自动截断较早内容

### 5.2 RAG 问答流程

```
用户提问
    │
    ▼
┌───────────────────────────────────────┐
│         第一步：并行检索                 │
│  ┌─────────────┐  ┌─────────────┐     │
│  │  知识库检索   │  │  知识图谱检索  │     │
│  │  (FAISS)    │  │  (图查询)   │     │
│  │  Top-K=5    │  │  实体邻居   │     │
│  └──────┬──────┘  └──────┬─────┘     │
└─────────┼────────────────┼───────────┘
          │                │
          ▼                ▼
┌───────────────────────────────────────┐
│        第二步：合并与排序               │
│   知识库检索结果优先                    │
│   知识图谱结果作为补充上下文             │
└──────────────────┬────────────────────┘
                   │
                   ▼
┌───────────────────────────────────────┐
│        第三步：构建 Prompt              │
│  · 系统提示词（含问答格式要求）          │
│  · 用户当前问题                        │
│  · 知识库检索片段（标注来源）            │
│  · 知识图谱子图信息                     │
│  · 对话历史上下文                      │
└──────────────────┬────────────────────┘
                   │
                   ▼
┌───────────────────────────────────────┐
│        第四步：Agent 生成回答           │
│   调用 DeepSeek 官方 API               │
│   生成结构化回答并附带引用来源           │
└──────────────────┬────────────────────┘
                   │
                   ▼
┌───────────────────────────────────────┐
│        第五步：结果展示                 │
│  · Agent 回答文本                      │
│  · 关联知识库片段（文档名+段落位置）     │
│  · 关联知识图谱子图（实体关系路径）      │
│  · 引用来源标注                        │
└───────────────────────────────────────┘
```

### 5.3 知识库与知识图谱的协同策略

| 维度 | 策略 |
|------|------|
| 检索优先级 | 知识库检索结果优先，知识图谱作为补充 |
| 并行方式 | 知识库检索和图谱检索同步进行，提升响应速度 |
| 结果合并 | 将图谱检索到的相关实体及其邻居实体作为额外上下文注入 Prompt |

### 5.4 回答结果展示

当用户提问能够在知识库或图谱中找到相关内容时，系统返回：

| 展示内容 | 说明 |
|----------|------|
| Agent 回答 | 基于检索结果生成的回答文本 |
| 知识库片段 | 匹配到的文档片段，标注来源文档名称和段落位置 |
| 知识图谱子图 | 相关实体在图谱中的关系路径，以文字描述或图片形式展示 |
| 来源引用 | 回答中明确标注参考了哪些文档的哪些段落 |

### 5.5 大模型配置

| 配置项 | 选择 |
|--------|------|
| 模型 | DeepSeek（第一期） |
| API 调用方式 | 官方 API 云端调用（`https://api.deepseek.com`） |
| 多模型切换 | **第一期不做**，后期版本实现 |

### 5.6 回答失败兜底策略

当知识库检索结果为空（Top-K 检索得分均低于阈值）或知识库为空时：

- Agent 主动提示用户：「当前知识库中未找到与您问题相关的内容，请先上传相关课程资料。」
- 不返回任何基于纯模型能力的「猜测性」回答，避免产生幻觉答案。
- 该提示信息内置于系统 Prompt 中，无需额外 API 调用。

### 5.7 扩展预留

| 功能 | 说明 |
|------|------|
| 置信度阈值控制 | 低于阈值的回答不返回或提示无法回答 |
| 多模型切换 | 支持在 DeepSeek、豆包等模型之间切换 |
| 回答质量评估 | 对回答进行自动评分或人工反馈收集 |

## 六、管理平台模块

### 6.1 功能范围

第一期管理平台为单用户本地 Web 应用，提供以下核心功能：

| 模块 | 功能点 |
|------|--------|
| 文档管理 | 文件上传、文档列表查看、删除文档、文档文本预览、全局知识库搜索 |
| 知识库管理 | 触发索引重建、查看索引状态、检索测试 |
| 图谱管理 | 触发图谱重建、下载图谱可视化图片、图谱节点详情查看 |
| Agent 问答 | 多轮对话界面、回答结果展示、对话历史管理（查看/重命名/删除）、用户反馈（有用/无用标记）|

### 6.2 功能详细说明

#### 6.2.1 全局知识库搜索

用户在管理平台中输入关键词，系统在已索引的知识库中检索匹配的文档片段，返回匹配的文档名称、段落位置和文本预览。

#### 6.2.2 文档文本预览

用户点击文档列表中的文档，可在线查看该文档的纯文本内容（解析后的文本），便于确认解析质量和内容。

#### 6.2.3 对话历史管理

系统自动保存用户与 Agent 的每次对话会话，支持：
- 查看历史会话列表（显示会话主题和创建时间）
- 重命名会话
- 删除会话
- 继续历史会话（Agent 恢复上下文继续对话）

会话历史数据存储于本地文件系统，格式为 JSON，每条记录包含会话 ID、主题、创建时间和对话内容列表。

#### 6.2.4 图谱节点详情

用户在图谱可视化图片中点击某个节点，系统展示该节点的详细信息：
- 实体名称
- 节点类型（实体/概念）
- 关联关系列表（邻居节点 + 关系类型）
- 来源 Chunk 列表（关联的文档片段）

#### 6.2.5 用户反馈机制

每次 Agent 回答后，用户可对回答进行「有用」或「无用」标记。反馈数据记录在本地文件中（JSON 格式），包含会话 ID、回答内容、反馈标签和时间戳，供后续版本进行回答质量分析和模型调优。

### 6.3 自动初始化流程

系统首次启动（用户首次访问 Web 页面）时，后端自动完成以下初始化操作：

1. 检测必要的目录结构是否存在，不存在则自动创建（`knowledge_base/`、`knowledge_graph/`、`dialogue_history/`、`user_feedback/` 等）
2. 检测 bge-m3 模型文件是否已下载，未下载则自动从 HuggingFace 下载至本地模型缓存目录
3. 加载配置文件（`config.yaml`），初始化各服务模块
4. 若已有历史数据（文档、索引、图谱），自动恢复到内存和索引中
5. 初始化完成后再响应前端请求

初始化过程中前端展示加载进度提示。

### 6.4 错误处理机制

| 场景 | 处理策略 |
|------|----------|
| 文档解析失败 | 跳过该文件，在索引重建结果中标注失败的文档及原因，前端提示用户 |
| API 调用超时 | 重试 1 次；若仍失败，返回错误信息「回答生成超时，请稍后重试」 |
| 模型 API 返回错误 | 记录错误日志，返回错误信息给前端，不展示给用户详细堆栈 |
| 索引文件损坏 | 提示用户触发全量索引重建 |
| 图谱构建失败 | 在图谱管理页面显示失败状态和错误摘要 |

### 6.5 技术选型

| 组件 | 选择 | 说明 |
|------|------|------|
| 后端框架 | FastAPI | Python Web 框架，轻量易用，支持异步 |
| 前端框架 | Vue 3 + Ant Design Vue | 主流前端框架 + 企业级 UI 组件库，生态成熟 |
| 页面形式 | Web 浏览器访问 | 跨平台，无需安装客户端 |

| 指标 | 参考值 | 说明 |

## 十一、附录

### 11.1 名词解释

| 术语 | 说明 |
|------|------|
| RAG | Retrieval-Augmented Generation，检索增强生成，通过外部知识库检索来增强大模型回答质量的技术 |

| bge-m3 | BAAI 开源的下一代文本向量嵌入模型，支持多语言和多功能检索 |
| DeepSeek | 国产大模型，第一期使用的主力语言模型，通过官方 API 调用 |
| Top-K | 向量检索时返回最相似的 K 个结果，默认 K=5 |
| SSE | Server-Sent Events，服务端推送技术，用于流式返回 Agent 回答 |
